# Phase 01 Plan 01.4: RLS Policies Summary

**One-liner:** Replaced 9 tenant-blind `authenticated_*` RLS policies with 17 org-scoped policies using `public.get_org_id()` across all 5 core CRM tables.

## What Was Built

Migration `20260331000004_rls_policies.sql` that:

1. **Drops 9 old blind policies** from `schema.sql` — policies that allowed any authenticated user to read any org's data
2. **Creates 17 new org-scoped policies** — enforcing `organization_id = public.get_org_id()` on every access

## Policy Count by Table

| Table         | Policies | Policy Names                                                                                        |
| ------------- | -------- | --------------------------------------------------------------------------------------------------- |
| contacts      | 4        | org_members_can_read, org_members_can_insert, org_members_can_update, managers_can_delete          |
| companies     | 4        | org_members_can_read, org_members_can_insert, org_members_can_update, managers_can_delete          |
| deals         | 4        | org_members_can_read, org_members_can_insert, org_members_can_update, managers_can_delete          |
| activities    | 4        | org_members_can_read, org_members_can_insert, org_members_can_update, managers_can_delete          |
| notifications | 1        | own_notifications (FOR ALL with user_id = auth.uid() AND organization_id = public.get_org_id())    |
| **Total**     | **17**   |                                                                                                     |

Note on the `auth.jwt()` check: `grep -c "auth.jwt()"` returns 1 because line 7 is a **comment** documenting the anti-pattern (`-- BAD: (auth.jwt() -> ...)`). No policy expression contains an inline JWT cast.

## Key Design Decisions

1. **No inline JWT casts** — all 17 policies call `public.get_org_id()` and `public.get_user_role()` rather than duplicating `(auth.jwt() -> 'app_metadata' ->> 'organization_id')::uuid`. The helper functions are O(1) and stable-cached per transaction.

2. **INSERT uses WITH CHECK, not USING** — PostgreSQL requires `WITH CHECK` for INSERT policies. Using `USING` on INSERT would silently fail or cause errors depending on version.

3. **DELETE restricted to admin/owner/manager** — `SELECT` and `UPDATE` are open to all org members; `DELETE` requires an elevated role. This matches the CRM's permission model where data deletion is a privileged operation.

4. **notifications upgraded to org-scoped** — The original `own_notifications` policy only checked `user_id = auth.uid()`. The replacement adds `AND organization_id = public.get_org_id()` to prevent cross-org notification leakage if a user ever belongs to multiple orgs.

## Post-Apply Verification (Manual — Supabase SQL Editor)

After applying this migration in the Supabase SQL Editor, run:

```sql
-- 1. Confirm old blind policies are gone
SELECT policyname FROM pg_policies
WHERE schemaname = 'public'
  AND policyname LIKE 'authenticated_%';
-- Expected: 0 rows

-- 2. Confirm org-scoped policies on contacts
SELECT policyname FROM pg_policies
WHERE schemaname = 'public' AND tablename = 'contacts'

--           org_members_can_read_contacts, org_members_can_update_contacts

  AND tablename IN ('contacts','companies','deals','activities','notifications');

```
