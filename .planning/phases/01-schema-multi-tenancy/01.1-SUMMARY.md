# Phase 01 Plan 1.1: Add organization_id FK to Core Tables Summary

**One-liner:** Added `organization_id uuid NOT NULL` with FK to `public.organizations(id) ON DELETE CASCADE` and a B-tree index on each of the 5 core CRM tables (contacts, companies, deals, activities, notifications).

## What Was Built

Migration `supabase/migrations/20260331000002_add_organization_id.sql` (Wave 2) alters the 5 existing CRM tables to carry a tenant-scoping column. Each table gains:

1. `TRUNCATE ... CASCADE` — clears dev data so PostgreSQL can accept `NOT NULL` without a DEFAULT value
2. `ALTER TABLE ... ADD COLUMN IF NOT EXISTS organization_id uuid NOT NULL REFERENCES public.organizations(id) ON DELETE CASCADE`
3. `CREATE INDEX IF NOT EXISTS idx_{table}_org_id ON public.{table}(organization_id)`

## Key Decision

**TRUNCATE approach for NOT NULL constraint**

PostgreSQL rejects `ADD COLUMN ... NOT NULL` when any existing rows lack a value for that column. Two options:

- Option A: `ADD COLUMN ... NOT NULL DEFAULT gen_random_uuid()` — adds a fake default that is meaningless (every row points to a non-existent org).
- Option B: `TRUNCATE ... CASCADE` then `ADD COLUMN ... NOT NULL` — valid because this is a dev database with no real user data.

Option B was chosen. It is cleaner, avoids a meaningless default, and was explicitly designed for in the research notes (Pitfall 1). The `WARNING` comment in the migration prevents accidental use against production data.
