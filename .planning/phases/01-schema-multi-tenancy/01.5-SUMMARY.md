## Task Results

### Task 1: Write migration 005 — JWT claim trigger

**Status: COMPLETE**

Created `supabase/migrations/20260331000005_jwt_trigger.sql` containing:

- `CREATE OR REPLACE FUNCTION public.handle_new_member()` with `SECURITY DEFINER` — required to write into `auth.users.raw_app_meta_data` from the `public` schema
- Two `PERFORM public.set_claim(...)` calls: one for `organization_id`, one for `user_role`
- `DROP TRIGGER IF EXISTS on_org_member_created ON public.organization_members` — idempotent drop before create
- `CREATE TRIGGER on_org_member_created AFTER INSERT OR UPDATE ON public.organization_members FOR EACH ROW EXECUTE FUNCTION public.handle_new_member()` — fires on both new memberships and role changes

**Important runtime note (Pitfall 3):** After this trigger fires, the client MUST call `supabase.auth.refreshSession()` to receive a new JWT. The cached JWT will not reflect new claims until refreshed.

The count 8 covers: contacts, companies, deals, activities, notifications (5 existing tables) + organization_members, invitations, gmail_tokens (3 new tables that reference organizations).

**Type correctness notes:**
- `organization_id` is NOT optional in Insert types for existing tables — SQL column is `NOT NULL` with no `DEFAULT`; callers must supply it

- `gmail_tokens.access_token` is `string | null` — matches nullable `text` column in migration 003
- `invitations.token` is `token?: string` in Insert — SQL generates via `DEFAULT encode(gen_random_bytes(32), 'hex')`
