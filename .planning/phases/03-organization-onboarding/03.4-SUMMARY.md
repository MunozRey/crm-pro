# Phase 03 Plan 03.4: Role JWT Trust Chain Summary

**One-liner:** Role trust chain completed: currentUser.role sourced from JWT app_metadata.user_role via normalizeRole helper with owner-to-admin aliasing

## What Was Built

Completed AUTH-09 and the UI-side portion of AUTH-10: the role used by `PermissionGate` and `ProtectedRoute` is now sourced from the server-controlled JWT `app_metadata.user_role` claim, not from the user-editable `user_metadata`.

### Changes

**Task 1 — normalizeRole helper + role source fix in authStore.ts (commit d103272e)**

Added a pure `normalizeRole()` function immediately before `initSupabaseAuth`. The function:
- Maps `'owner'` to `'admin'` (defensive coverage for Supabase org-creator alias)
- Accepts any of the four canonical roles unchanged
- Falls back to `'sales_rep'` for any unrecognized string (least-privilege default)

Changed the role assignment inside `onAuthStateChange`:
```typescript
// Before (user-editable user_metadata)
role: (sbUser.user_metadata?.role ?? 'sales_rep') as UserRole,

// After (server-controlled app_metadata, with user_metadata fallback)
role: normalizeRole(sbUser.app_metadata?.user_role ?? sbUser.user_metadata?.role),
```

The mock mode path (`isSupabaseConfigured = false` / SEED_USERS) is untouched — it never reaches `onAuthStateChange`.

**Task 2 — VALID_ROLES export in permissions.ts (commit 265306df)**

Added `VALID_ROLES` as a readonly exported constant after `ROLE_PERMISSIONS`, with a JSDoc comment documenting that `'owner'` is normalized before reaching this map. This documents the normalization contract at the permissions layer boundary for future consumers.

`PermissionGate.tsx` and `ProtectedRoute.tsx` required no code changes — they already read `currentUser.role` via `useAuthStore` selectors. Verified both files contain zero hardcoded role strings.

### Trust Chain

The full role enforcement chain is now:
1. Phase 1 DB trigger writes `user_role` into `app_metadata` on `organization_members` insert/update
2. `initSupabaseAuth.onAuthStateChange` reads `app_metadata.user_role` → runs through `normalizeRole()` → sets `currentUser.role`
3. `PermissionGate` and `ProtectedRoute` read `currentUser.role` via Zustand selectors → call `hasPermission()` → enforce UI access control
4. RLS policies at the database layer enforce the same `user_role` claim independently
