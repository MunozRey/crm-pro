# Phase 03 Plan 03.1: Organization Onboarding Flow Summary

**One-liner:** First-login org creation flow: OrgSetup page with Supabase insert + JWT refresh, gated by ProtectedRoute organizationId guard

## What Was Built

Implemented AUTH-06: the first-login organization creation flow that gates the entire CRM behind org membership.

### Flow

1. New user completes email verification and logs in
2. Their JWT has no `organization_id` in `app_metadata` — `currentUser.organizationId` is `undefined`
3. `ProtectedRoute` detects `organizationId` is falsy (and `isSupabaseConfigured` is true) → redirects to `/org-setup`
4. User fills out org name and slug on the `OrgSetup` page
5. `handleSubmit` inserts into `organizations`, then `organization_members` (role: admin), then calls `supabase.auth.refreshSession()`
6. `onAuthStateChange` fires in `authStore`, updates `currentUser.organizationId` with the new org ID from the refreshed JWT
7. `navigate('/', { replace: true })` — `ProtectedRoute` now passes, user reaches the dashboard

### Demo mode preserved

The `isSupabaseConfigured` gate in `ProtectedRoute` ensures mock/demo users (no Supabase configured) are never redirected to `/org-setup`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] TypeScript type error: Supabase insert payload rejected as `never`**
- **Found during:** Task 3
- **Issue:** `database.types.ts` Insert types for `organizations` and `organization_members` omit only `created_at`/`joined_at`, leaving `id` required. The Supabase `insert()` call with just `{ name, domain }` caused TS2769 overload mismatch.
- **Fix:** Used `as any` cast on the Supabase client reference for insert calls. Added inline type assertions for error and org data access. This avoids modifying `database.types.ts` (out of scope) while keeping runtime behavior correct — the DB auto-generates UUIDs.
- **Files modified:** `src/pages/OrgSetup.tsx`
- **Commit:** 76defea9
