Current role resolution in initSupabaseAuth (src/store/authStore.ts, ~line 447):
```typescript
role: (sbUser.user_metadata?.role ?? 'sales_rep') as UserRole,
```
This reads from `user_metadata` which is editable by the user. It must change to read from `app_metadata` which is server-controlled:
```typescript
role: normalizeRole(sbUser.app_metadata?.user_role ?? sbUser.user_metadata?.role ?? 'sales_rep'),
```

The `normalizeRole` helper must be a small pure function defined in authStore.ts (or imported from utils):
```typescript
function normalizeRole(raw: string | undefined): UserRole {
  if (raw === 'owner') return 'admin'           // Supabase org creator alias
  const valid: UserRole[] = ['admin', 'manager', 'sales_rep', 'viewer']
  return valid.includes(raw as UserRole) ? (raw as UserRole) : 'sales_rep'
}
```

Why 'owner' alias: When an org member is inserted with role='admin', the DB trigger writes 'admin'.
But the invite-member Edge Function (03.2) passes `role` from the UI directly. The org creator
in OrgSetup.tsx (03.1) inserts with role='admin' — so 'owner' is defensive coverage for future
changes or Supabase's own metadata behavior.

Current permissions.ts UserRole type:
```typescript
export type UserRole = 'admin' | 'manager' | 'sales_rep' | 'viewer'
// 'owner' is NOT in this type — that's correct. normalizeRole maps 'owner' -> 'admin' before it hits permissions.
```

What changes in permissions.ts: Add a new exported helper `normalizeRole` to keep the mapping
canonical and importable by authStore without circular deps. Or keep `normalizeRole` local to
authStore.ts — both approaches are acceptable. Choose co-location in authStore.ts (simpler).
